### 1️⃣ 핵심 개념 한 줄
> 면접에서 첫 문장으로 바로 말할 수 있는 정의 한 문장 <br>
> Spring DI 란 객체 생성과 의존성 연결 책임을 애플리케이션 코드가 아닌  
> Spring IoC 컨테이너가 담당하도록 하여 클래스는 필요한 인터페이스만 의존하고  
> 실제 구현 객체는 외부에서 주입받도록하는 설계입니다.  
> 이를 통해 클래스간 결합도를 낮추고 테스트, 교체, 확장이 용이한 구조를 만들 수 있습니다

---

### 2️⃣ 동작 원리 요약

- **왜 필요한가?**
  - DI 이전에는 클래스가 필요한 객체를 직접 생성해서 사용하고 있었고 이런 경우
  - 코드에 해당 객체가 강하게 결합되어있어 테스트, 변경 등이 복잡하고 많은 코드 수정이 이루어져야 했음  
  - 이러한 문제를 해결하고자 DI 를 사용하여 객체를 받아오면 교체시 인터페이스만 바꿔주면 되어 교체가 편해지고  
  - 테스트 시에도 외부 객체를 모킹하여 독립적인 테스트도 더 간단하게 할 수 있다
- **어떻게 동작하는가?**
  - Bean 생성과 DI 는 애플리케이션 실행 시점(Runtime) 에 IoC 컨테이너 초기화 과정에서 이루어진다  
  - 우선 애플리케이션이 시작되면 application context 가 생성되고, IoC 컨테이너가 시작된다  
  - 이후 BeanDefinition 이 수집된다 @Component, @Service, @Repository, @Bean, @Configuration 을 전부 스캔해서 목록을 생성한다 (객체 생성 X)(여기에는 클래스 타입, 스코프, 생성자 정보, 의존성 정보가 포함된다)  
  - 이제 컨테이너에서 BeanDefinition 기반으로 객체를 생성한다 (생성자는 호출 되지만 아직 의존성은 안 들어 있을 수도 있음)  
  - 이제 의존성을 주입해서 객체 그래프를 완성 시킨다  
  - 이후 AOP 대상이면 프록시를 생성해서 원본 Bean 을 감싸고 최종 Bean 을 교체등록한다  
  - 컨테이너에는 k: Bean 이름 / 타입, v: 실제 객체 (프록시일 수도 있음)  
  - 이후 요청시 이미 만들어진 Bean 을 반환한다 (싱글톤)

### 3️⃣ 핵심 키워드

- `DI 방식`
- `Bean 생명주기`
- `무엇을 위한 IoC 인가`
- 'Bean Scope'
  - 각 Bean 마다 사용용도가 다를 수 있기 때문에 해당 용도별 Scope 존재
  - singleton: 컨테이너당 1개 가장 기본이 되는 사용
  - prototype: 요청마다 새 객체
  - request: HTTP 요청당 1개
  - session: HTTP 세션당 1개
  - application: 웹 앱 전체 1개
  - websocket: 웹소켓 세션당 1개
